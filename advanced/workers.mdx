Workers enable multi-process architecture for your Discord bot, distributing shards across separate worker processes. This provides better CPU utilization, fault isolation, and scalability compared to single-process sharding.

## Overview

Seyfert's worker system consists of:

- **WorkerManager** - Main process that coordinates workers and handles IPC
- **WorkerClient** - Client running in each worker process
- **IPC Communication** - Message passing between manager and workers

Supported worker modes:
- `threads` - Node.js Worker Threads
- `clusters` - Node.js Cluster (process forking)
- `custom` - Custom adapter for alternative environments

## Architecture

```
┌─────────────────┐
│  WorkerManager  │ (Main Process)
│  - Coordinates  │
│  - Handles API  │
│  - Manages IPC  │
└────────┬────────┘
         │
    ┌────┴────┬────────┬────────┐
    │         │        │        │
┌───▼───┐ ┌──▼───┐ ┌──▼───┐ ┌──▼───┐
│Worker0│ │Worker1│ │Worker2│ │Worker3│
│Shard  │ │Shard  │ │Shard  │ │Shard  │
│ 0-2   │ │ 3-5   │ │ 6-8   │ │ 9-11 │
└───────┘ └───────┘ └───────┘ └───────┘
```

## Quick Start

### Main Process (manager.ts)

```typescript
import { WorkerManager } from 'seyfert';
import { GatewayIntentBits } from 'seyfert/types';
import { resolve } from 'path';

const manager = new WorkerManager({
  token: process.env.DISCORD_TOKEN!,
  intents: GatewayIntentBits.Guilds | GatewayIntentBits.GuildMessages,
  info: await fetch('https://discord.com/api/v10/gateway/bot', {
    headers: { Authorization: `Bot ${process.env.DISCORD_TOKEN}` }
  }).then(r => r.json()),
  
  // Path to worker file
  path: resolve('./worker.js'),
  
  // Worker mode
  mode: 'threads',
  
  // Number of shards per worker
  shardsPerWorker: 3,
  
  // Optional: total number of workers (auto-calculated if omitted)
  workers: 4,
  
  // Handle gateway events from all workers
  handlePayload: async (shardId, workerId, payload) => {
    console.log(`Worker ${workerId}, Shard ${shardId}:`, payload.t);
  }
});

await manager.start();
console.log('All workers started!');
```

<ParamField path="token" type="string">

<ParamField path="intents" type="number">

<ParamField path="workers" type="number">

<ParamField path="shardStart" type="number" default="0">

</ParamField>

<ParamField path="shardEnd" type="number" default="totalShards">
  Last shard ID (for distributed setups)
</ParamField>

<ParamField path="workerProxy" type="boolean" default="false">

</ParamField>

<ParamField path="debug" type="boolean" default="false">

</ParamField>

<ParamField path="compress" type="boolean" default="false">

</ParamField>

<ParamField path="handlePayload" type="function">

## WorkerManager Methods

### Worker Communication

```typescript
// Execute code on a specific worker
const result = await manager.tellWorker(
  0, // worker ID
  (client, vars) => {
    // This runs in the worker process
    return {
      shardCount: client.shards.size,
      userId: vars.userId
    };
  },
  { userId: '123456789' } // variables passed to the function

// Execute code on all workers
const results = await manager.tellWorkers(

    return client.guilds.cache.count('guild');
```

## WorkerClient API

The `WorkerClient` extends `BaseClient` with worker-specific features.

### Worker-Specific Methods

```typescript
import { WorkerClient } from 'seyfert';

const client = new WorkerClient();

// Access worker metadata
console.log(`Worker ID: ${client.workerId}`);
console.log(`Managing shards:`, client.workerData.shards);
console.log(`Total shards: ${client.workerData.totalShards}`);

// Calculate shard for a guild
const shardId = client.calculateShardId('123456789012345678');

// Access individual shards
for (const [id, shard] of client.shards) {
  console.log(`Shard ${id}: ${shard.latency}ms`);
}

// Average latency across this worker's shards
console.log(`Worker latency: ${client.latency}ms`);
```

### Worker Events

Workers emit special lifecycle events:

```typescript
import { createEvent } from 'seyfert';

// When all shards in this worker are connected
export default createEvent({
  data: { name: 'WORKER_SHARDS_CONNECTED' },
  run: (user, client, shardId) => {
    console.log(`Worker ${client.workerId} shards connected`);
  }
});

// When all shards in this worker are ready
export default createEvent({
  data: { name: 'WORKER_READY' },
  run: (user, client, shardId) => {
    console.log(`Worker ${client.workerId} is ready`);
  }
});

// When ALL workers are ready (fired on first worker only)
export default createEvent({
  data: { name: 'BOT_READY' },
  run: (user, client, shardId) => {
    console.log('Entire bot is ready!');
  }
});
```

### API Request Proxying

Prevent rate limit issues by routing all API requests through the manager:

```typescript
const manager = new WorkerManager({
  // ... options
  workerProxy: true // Enable API proxying
});
```

With `workerProxy: true`, workers send API requests to the manager via IPC instead of making direct HTTP calls. The manager handles rate limiting centrally.

## IPC Message Flow

The manager and workers communicate via typed IPC messages:

**Manager → Worker:**
- `SPAWN_SHARDS` - Initialize shards
- `ALLOW_CONNECT` - Permission to connect a shard
- `SEND_PAYLOAD` - Send gateway payload
- `SHARD_INFO` / `WORKER_INFO` - Request shard data
- `EXECUTE_EVAL` - Run code in worker
- `HEARTBEAT` - Health check

**Worker → Manager:**
- `WORKER_START` - Worker initialized
- `WORKER_READY` - All shards ready
- `CONNECT_QUEUE` - Request to connect shard
- `RECEIVE_PAYLOAD` - Forward gateway event
- `CACHE_REQUEST` - Request cache operation
- `WORKER_API_REQUEST` - Proxied API request
- `ACK_HEARTBEAT` - Heartbeat response

## Best Practices

<AccordionGroup>
  <Accordion title="Choose the right shardsPerWorker">
    - **Small bots (< 1,000 guilds):** 1-2 workers with all shards
    - **Medium bots (1,000-10,000 guilds):** 3-5 shards per worker
    - **Large bots (10,000+ guilds):** 2-3 shards per worker
    
    More workers = better CPU utilization but more overhead
  </Accordion>
  
  <Accordion title="Use workerProxy for large bots">
    Enable `workerProxy: true` to centralize rate limiting and prevent 429 errors when workers make many API requests.
  </Accordion>
  
  <Accordion title="Monitor worker health">
    The built-in heartbeat system handles worker crashes, but consider logging worker restarts:
    
    ```typescript
    handleWorkerMessage: (message) => {
      if (message.type === 'WORKER_READY') {
        console.log(`Worker ${message.workerId} (re)started`);
      }
    }
    ```
  </Accordion>
  
  <Accordion title="Handle worker-specific state carefully">
    Remember that each worker has its own memory space. Avoid storing critical state in-memory - use shared cache or databases instead.
  </Accordion>
</AccordionGroup>

## Comparison: Workers vs Sharding

| Feature | ShardManager | WorkerManager |
|---------|-------------|---------------|
| **Process Model** | Single process | Multi-process |
| **CPU Utilization** | Single core | Multiple cores |
| **Memory Isolation** | Shared memory | Isolated per worker |
| **Fault Tolerance** | Shard crash affects all | Worker crash isolated |
| **Complexity** | Lower | Higher |
| **IPC Overhead** | None | Message passing |
| **Best For** | Small-medium bots | Large-scale bots |

<Info>
  For most bots under 10,000 guilds, `ShardManager` is simpler and sufficient. Use `WorkerManager` when you need better CPU utilization or fault isolation.
</Info>
